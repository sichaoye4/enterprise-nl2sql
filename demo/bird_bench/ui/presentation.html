Architecture Briefing

Enterprise NL2SQL

Semantic Registry & Pattern Memory System

Converting business questions into accurate SQL with enriched schema context, learned query patterns, and execution-guided repair.

Problem Statement

Enterprise NL2SQL fails most often when natural-language intent has to be grounded in physical schemas, domain values, and multi-table logic.

Column mismatch

Business terms rarely match physical column names one-to-one, especially with abbreviated or domain-specific schemas.

Join complexity

Analytical questions often require 3+ tables, exact foreign-key paths, and careful result grain.

Filter values

Models shorten labels or infer literals instead of copying exact categorical values from the database.

Ranking semantics

Superlatives and top-N questions require correct ORDER BY direction plus LIMIT.

Aggregation rules

WHERE vs HAVING, ratio math, DISTINCT, CASE, and subqueries create subtle failure modes.

Production feedback

Without repair and memory, near-misses are not converted into durable system improvements.

Architecture Overview

The system routes questions, builds grounded prompts from schema and memory, generates SQL with DeepSeek, then validates and repairs before returning results.

Schema Context

Raw DDL is preserved, then augmented with the metadata the model needs to map business language to executable SQL.

DDL + column metadata

SQLite CREATE TABLE text, data types, primary keys, foreign keys, and qualified column ownership.

Descriptions

Database description CSVs provide human-readable table and column documentation.

Join paths

Explicit PRAGMA foreign keys plus inferred paths for common ID and code patterns.

Sample values

Top distinct text values ground categorical filters and prevent shortened literals.

Semantic hints

Term-to-concept-to-metric mappings from bird_semantic point to physical expressions.

Evidence passthrough

BIRD evidence is retained alongside schema context for formulas and domain definitions.

Execution Repair Loop

The repair path catches syntax failures, empty results, missing constructs, and static SQL risks before choosing the best attempt.

01 Generate

Initial SQL

Prompt includes schema context, evidence, and retrieved few-shot examples.

02 Validate

Static checks

Unknown columns, WHERE/HAVING, ranking constructs, and integer division.

03 Execute

SQLite run

Captures runtime errors and result metadata such as row count.

04 Repair

Targeted retry

Builds a focused fix prompt with failed SQL and error reason, up to 2 retries.

76%19 of 25 repair attempts corrected the SQL in observed runs.

Best-attempt selection

Prefers valid, executable, non-empty SQL that satisfies required constructs.

Pattern Memory

Pattern memory turns past successful question-SQL pairs into few-shot examples, with V2 expanding from regex labels to semantic retrieval.

V1: Rule-Based

  • SQLite-backed question and SQL store
  • Regex classifier across 9 query types
  • Retrieval by same database, query type, and keyword overlap
  • Seeded from 1,533 BIRD dev patterns

V2: LLM-Powered

  • Extracts business context, technical pattern, and key entities
  • Stores rich metadata: category, metrics, dimensions, and tags
  • Searches by entity overlap, technical pattern, same DB, and rerank
  • V2.5 roadmap adds SQL AST and schema-footprint matching
